Loxia en 30 secondes
Le projet
Une plateforme de location immobilière qui met en relation propriétaires et locataires, bâtie sur une vraie architecture distribuée.
Un seul compte
Tout utilisateur peut à la fois publier des annonces et candidater à des locations. Le basculement se fait via un simple toggle dans l'interface.
Objectif pédagogique
Démontrer une architecture microservices réaliste — sans sur-ingénierie, déployable en une commande.

02 · Architecture
02 · Architecture
Les services se parlent entre eux
🏠 → 📋   Annonce verrouillée ?
Avant de modifier ou supprimer une annonce, catalog demande à rental si elle a une candidature en cours. Si oui → refus.
🏠 → 🔑   Qui est le propriétaire ?
Quand on consulte une annonce, catalog demande à auth le nom du propriétaire pour l'afficher dans la fiche.
📋 → 🔔   Envoyer une notification
Quand une candidature est acceptée ou refusée, rental demande à notification d'envoyer une notification au locataire.
Si un service ne répond pas ?
Deux comportements selon l'enjeu :

Bloquant — vérification du lock : on refuse par précaution si rental est indisponible.

Optionnel — nom du propriétaire, notification : l'action réussit quand même, sans les données secondaires.
Principe de base
Chaque service reste indépendant — il n'accède jamais directement à la base de données d'un autre. Tout passe par des appels entre services.
03 · Sécurité
03 · Sécurité
Un seul point de sécurité
1
L'utilisateur se connecte → auth-service génère un token signé
2
Le token est inclus dans chaque requête vers l'API Gateway
3
La gateway vérifie le token, puis transmet les infos utilisateur aux services
4
Les services reçoivent X-User-Id et X-User-Email — sans re-vérifier
Token de courte durée · 15 min
L'accès expire vite. Un token de renouvellement (7 jours) permet de reconnecter automatiquement l'utilisateur, sans qu'il s'en aperçoive.
Les services internes sont invisibles
Seule la gateway est accessible depuis l'extérieur. Toutes les routes /internal/** sont bloquées — elles n'existent que pour les appels entre services.
Renouvellement transparent
Le frontend renouvelle le token automatiquement en arrière-plan. L'utilisateur reste connecté sans interruption.

04 · Fonctionnalités
04 · Règle métier clé
L'annonce verrouillée
📌 Une annonce avec une candidature EN ATTENTE ou ACCEPTÉE ne peut plus être modifiée ni supprimée.
🏠
catalog-service
PUT /api/catalog/listings/{id}
DELETE /api/catalog/listings/{id}
↓ vérifie le lock avant d'agir
↓ 409 Conflict si verrouillé
→ Est-ce que cette annonce est verrouillée ?
⇄
← oui / non
📋
rental-service
SELECT COUNT(*) > 0 FROM applications WHERE listing_id = ? AND status IN ('PENDING', 'ACCEPTED')
Si le service ne répond pas
La modification est bloquée par précaution. Mieux vaut refuser que corrompre les données.
Vérification en lot
Sur les listes, une seule requête vérifie tous les locks d'un coup — pas de surcharge réseau.

Conclusion
Conclusion
Ce que Loxia démontre
🏗️ Architecture distribuée réelle
5 services indépendants, chacun avec sa propre base — isolation totale des données.
🔒 Sécurité centralisée
Un seul point de validation JWT. Les services internes ne sont jamais exposés.
📐 Bonnes pratiques Java
DTOs séparés, injection par constructeur, migrations Flyway, tests unitaires.
⚛️ Frontend moderne
React 18, gestion du cache serveur, reconnexion automatique, formulaires validés.
🐳 Déployable en une commande
Docker Compose orchestre les 8 conteneurs avec démarrage ordonné et données de démo.
📚 Entièrement documenté
Architecture, décisions techniques, maquettes, script de démo — prêt à onboarder.
